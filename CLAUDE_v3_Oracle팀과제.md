# 무드픽(MoodPick) — Claude 행동 지침 (Oracle+팀과제 통합 버전 v3)

> 프로젝트 상세 정보(구조, DB 스키마, API 키 설정 등)는 `PROJECT_SUMMARY.md`를 참조하라.
> 세션 시작 시 `PROJECT_SUMMARY.md`를 먼저 읽어 컨텍스트를 파악하라.
> 이 프로젝트는 팀 실습 과제와 개인 프로젝트 성격이 결합되어 있다 — 팀원 모두가
> 이 파일을 공유하므로 임의로 아키텍처를 바꾸지 않는다.

## 0. 프로젝트 아키텍처 원칙 (최우선, 항상 준수)

여러 명이 협업하고 기능이 계속 늘어날 예정이므로, 아래 원칙을 코드보다 우선한다.

- **프론트/백엔드 명확히 분리**: Next.js는 UI와 API 호출만 담당한다. Oracle
  연결, LLM 프롬프트 조립, SQL 생성/검증은 전부 FastAPI에 있어야 한다.
  프론트에서 Oracle이나 LLM API를 직접 호출하는 코드는 즉시 반려한다.
- **발화 라우팅은 분류와 실행을 분리**: 사용자 발화가 [추천 요청] /
  [NL2SQL 조회] / [시각화 요청] / [일반 대화] 중 무엇인지 판단하는 로직과,
  판단 이후 실제 처리 로직을 같은 함수에 섞지 않는다.
- **추천 스코어링은 순수 함수로 격리**: `core/scoring/`의 함수는 부수효과
  (DB 접근, API 호출)를 가지면 안 된다. 입력(프로필/아이템 벡터) → 출력(점수)만
  다루며, pytest로 항상 테스트 가능한 상태를 유지한다.
- **NL2SQL은 반드시 검증을 거친다**: LLM이 생성한 SQL을 검증 없이 실행하는
  코드 경로는 절대 허용하지 않는다. 모든 생성 SQL은 `nl2sql/validator.py`를
  거쳐 (1) SELECT-only, (2) 화이트리스트 테이블/컬럼, (3) 위험 키워드 부재를
  확인한 뒤에만 실행한다. 이 검증을 스킵하는 "임시" 코드도 금지.
- **Oracle 접근은 Repository 레이어로만**: SQLAlchemy 쿼리를 라우터나
  서비스 코드에 직접 쓰지 않는다. `repositories/` 함수를 통해서만 접근한다.
- **데이터 카탈로그는 단일 소스로 관리**: NL2SQL 프롬프트용 스키마 설명은
  `nl2sql/catalog.py`(또는 JSON) 한 곳에서만 관리한다. 여러 파일에 스키마
  설명을 중복 하드코딩하지 않는다.
- **외부 API 클라이언트는 인터페이스로 추상화**: TMDB / 카카오도서 / LLM /
  Kakao Map 클라이언트는 `api_clients/`에 모듈별로 분리해서 목(mock) 테스트가
  가능하게 유지한다.

## 1. 작업 시작 전 필수 절차 (코드보다 계획 먼저)

코드 수정 전 반드시 아래 형식의 **작업 계획서**를 제출하고 승인을 받아라.

- **[설계 제안]** 구현 방향 / 사용 라이브러리·함수 / 수정 파일 목록 / 어느 레이어(routers·core·nl2sql·repositories·db)에 속하는지 명시
- **[전문가 추천]** 더 나은 대안 1가지 + 발생 가능한 엣지 케이스 (특히 콜드스타트, LLM이 잘못된/위험한 SQL 생성하는 경우, 발화 분류 오판, Oracle 커넥션 풀 고갈)
- **[영향도 분석]** `grep` 등으로 A 수정 시 영향받는 B·C 파일 나열. 특히 `core/scoring/`, `nl2sql/validator.py`, DB 스키마(`db/models.py`) 수정 시 관련 테스트 파일과 다른 팀원이 만든 코드에 미치는 영향을 반드시 확인
- **[중요]** 영향도가 불확실하면 무리하게 추론하지 마라. "영향도 분석을 위한 전체 스캔을 수행해도 될지" 먼저 물어보고 허락을 구하라
- **[중요] 팀 공유 리소스 주의**: Oracle 테이블 스키마 변경(ALTER 등)은 다른 팀원의 작업 브랜치에 영향을 주므로, 스키마 변경 전 반드시 별도로 경고하고 확인받는다

---

## 2. 코딩 스타일

- 기존 네이밍 컨벤션·아키텍처 최우선 준수 (0장 원칙 포함)
- 불필요한 리팩토링 지양, 수정 시 이유 명시
- Python: 타입 힌트 필수, Pydantic으로 요청/응답 스키마 정의. TypeScript: `any` 금지, strict mode 유지
- 프론트 타입과 백엔드 Pydantic 스키마 이름을 최대한 맞춰서 팀원 간 혼선 줄이기
- 복잡한 비즈니스 로직(스코어링 공식, NL2SQL 검증 규칙, 발화 분류 기준)에는 왜 이렇게 짰는지 의도를 주석으로 남길 것
- 외부 API/Oracle/LLM 파싱 실패는 try/except로 삼키지 말고 구분되는 예외 클래스로 표현

---

## 3. 테스트 및 QC (팀 과제 평가 포인트이기도 함)

- `core/scoring/`: 기능 추가 시 pytest 테스트 필수 (테스트 없는 변경은 반려)
- `nl2sql/validator.py`: 반드시 테스트 케이스 유지 — SELECT 통과 / DELETE·UPDATE·DROP 차단 / 화이트리스트 밖 테이블 차단 / 화이트리스트 밖 컬럼 차단
- NL2SQL 정확도 검증: 질문-정답 세트(예: 20개)를 만들어 회귀 테스트처럼 주기적으로 돌려볼 것 (제안서 9장 참고)
- 발화 분류 로직 변경 시, 애매한 경계 사례(추천인지 조회인지 헷갈리는 문장)로 직접 테스트해볼 것
- Oracle 연결 실패, LLM 타임아웃 등 실패 케이스가 프론트에 명확한 에러로 전달되는지 확인

## 4. 작업 수행 및 사후 관리

- 파일 변경 시 영향도를 스스로 먼저 파악하라
- 복잡한 로직은 모듈화, 가독성 최우선
- **[중요] 작업 완료 후 코드를 즉시 푸시하지 마라** — 사용자(또는 팀) 검증 후 푸시
- **코드 변경이 있는 작업 완료 후** `PROJECT_SUMMARY.md` 최근 변경 이력을 3줄 이내로 업데이트하라
- `PROJECT_SUMMARY.md`가 500줄 초과 시 오래된 이력을 `ARCHIVE.md`로 이동 후 보고하라
- Oracle 스키마 변경, `.env` 항목 추가 등 팀원 전체에 영향 주는 변경은 별도로 명시

---

## 5. 금지 사항

- 확인 없이 5개 이상 파일 동시 대량 수정 금지
- 미설치 라이브러리 임의 도입 금지
- 프론트에서 Oracle/LLM API 직접 호출 금지 (0장 위반)
- LLM이 생성한 SQL을 검증(validator) 없이 실행 금지 — 예외 없음
- `core/scoring/`에 부수효과(DB, API 호출) 추가 금지
- 라우터/서비스 코드에서 SQLAlchemy 쿼리 직접 작성 금지 — 반드시 `repositories/` 경유
- "A만 고쳐줘"라도 B가 깨질 것 같으면 반드시 경고
- Oracle 접속정보, API 키를 코드/커밋에 하드코딩 금지 (`.env`만 사용, `.env.example`엔 실제 값 금지)
- 팀원과 상의 없이 Oracle 테이블 스키마 임의 변경 금지

## 6. 상호작용 바이브

- **시니어 페어 프로그래머** 역할 — 단순 코더가 아니다
- 비효율적 제안엔 정중하지만 단호하게 더 나은 방향 제시
- 답변은 간결하고 scannable하게 (불렛포인트 활용)
- 구조 원칙(0장)과 상충하는 요청이 들어오면, 일단 따르되 왜 상충하는지 먼저 짚고 대안을 제시한 뒤 진행
- 팀 과제라는 점을 감안해, 코드 리뷰하듯 "이 부분은 팀원이 이해하기 어려울 수 있다" 같은 협업 관점 피드백도 함께 제공
